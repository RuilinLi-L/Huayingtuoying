# 华音拓影内容准备规范

本文档用于规范项目后续素材与内容的准备方式，确保内容团队、建模团队、音频团队与技术团队能够按统一标准交付。  
当前规范基于一期现有框架制定，目标是：**素材一旦备齐，即可直接接入当前系统，不需要大幅修改代码结构。**

---

## 1. 适用范围

本规范适用于以下内容：

- 乐器主题条目内容
- 校园叙事与乐理知识内容
- AR 识别图与展示海报
- 3D 模型与贴图
- 分轨音频
- NFC / 二维码入口内容

当前框架默认能力边界如下：

- 1 个条目对应 1 个主 AR 模型
- 1 个条目对应 1 套识别图资源
- 1 个条目对应多条音频分轨
- 1 个条目对应多张知识卡
- 当前 AR 默认展示前 3 个热点

如果后续要扩展为多模型舞台、更多热点、多人站位等复杂能力，内容规范仍可复用，但技术接入方式会进一步扩展。

---

## 2. 内容交付总原则

### 2.1 一条内容 = 一个完整条目包

每个条目建议作为一个完整内容包准备，至少包括：

- 条目基础信息
- 海报图
- 识别图
- `.mind` 识别文件
- 主模型
- 分轨音频
- 知识卡
- NFC / 二维码入口映射信息

### 2.2 命名统一

所有资源都以 `entryId` 为核心标识，保持一一对应。

示例：

- `violin-dialogue`
- `flute-color`
- `ensemble-stage`

建议规则：

- 全小写
- 单词之间使用短横线 `-`
- 不使用空格
- 不使用中文
- 不使用特殊符号

### 2.3 所有正式内容必须可替换占位素材

当前系统中已有占位素材用于演示，但正式阶段必须以正式资源替换。  
正式内容应满足：

- 路径明确
- 命名统一
- 文件格式稳定
- 可直接放入 `public/assets/...`

## 3. 条目配置规范

条目配置统一维护在：

`src/data/entries.json`

相关数据结构定义在：

`src/types/manifest.ts`

每个条目都必须满足以下字段要求。

## 3.1 基础字段

### `id`

- 类型：`string`
- 必填：是
- 说明：条目唯一标识，用于 URL、二维码、NFC、资源命名
- 格式要求：
  - 小写英文
  - 推荐 2–4 个词
  - 使用短横线连接

示例：

```json
"id": "cello-memory"
```

### `title`

- 类型：`string`
- 必填：是
- 说明：条目主标题
- 建议长度：4–12 个中文字符

### `subtitle`

- 类型：`string`
- 必填：建议填写
- 说明：简要说明条目主题
- 建议长度：12–30 个中文字符

### `description`

- 类型：`string`
- 必填：建议填写
- 说明：条目介绍文案，用于条目页和体验页
- 建议长度：40–120 个中文字符

### `orchestraZone`

- 类型：`string`
- 必填：建议填写
- 说明：所属乐器分组
- 推荐值：
  - `弦乐组`
  - `木管组`
  - `铜管组`
  - `打击乐组`
  - `键盘组`
  - `全编制示例`

### `themeColor`

- 类型：`string`
- 必填：建议填写
- 说明：条目主题色
- 格式：十六进制颜色值

示例：

```json
"themeColor": "#8b5cf6"
```

### `sceneType`

- 类型：`string`
- 当前固定值：`mindar-image`

### `fallbackMode`

- 类型：`string`
- 当前可选值：
  - `poster`
  - `model`

建议：

- 单乐器条目优先使用 `poster`
- 舞台类或综合类条目可考虑 `model`

## 3.2 推荐条目模板

```json
{
  "id": "cello-memory",
  "title": "大提琴示例",
  "subtitle": "以中低音弦乐展示厚度与歌唱性",
  "description": "通过主旋律、和声与节奏层的切换，理解大提琴在乐队中的情感表达和结构支撑作用。",
  "orchestraZone": "弦乐组",
  "themeColor": "#7c3aed",
  "sceneType": "mindar-image",
  "targetImage": "/assets/markers/cello-memory.png",
  "trackingTargetSrc": "/assets/markers/cello-memory.mind",
  "posterImage": "/assets/posters/cello-memory.svg",
  "modelUrl": "/assets/models/cello-memory/scene.glb",
  "fallbackMode": "poster",
  "audioStems": [],
  "knowledgeCards": []
}
```

## 4. 视觉素材规范

视觉素材分为两类：

- 展示海报 `posterImage`
- 识别图 `targetImage`

## 4.1 展示海报规范

### 用途

- 条目入口页展示
- 非 AR 环境下的 2D fallback 展示
- 条目卡片封面展示

### 存放路径

建议存放到：

`public/assets/posters/`

### 文件格式

推荐优先级：

1. `svg`
2. `png`
3. `jpg`

### 命名规范

使用 `entryId` 命名：

- `violin-dialogue.svg`
- `flute-color.svg`
- `ensemble-stage.svg`

### 画面要求

- 保证主题突出
- 与条目标题一致
- 保持整体风格统一
- 尽量与校园 IP 叙事一致
- 保证移动端查看可读

### 建议尺寸

- 推荐比例：`16:10` 或接近当前演示比例
- 推荐宽度：`1200px` 以上

## 4.2 识别图规范

### 用途

- AR 图像识别的视觉目标
- 用户实际对准扫描的实体图样

### 存放路径

建议存放到：

`public/assets/markers/`

### 文件格式

- `png` 优先

### 命名规范

- `entryId.png`
- `entryId.mind`

例如：

- `cello-memory.png`
- `cello-memory.mind`

### 内容设计要求

识别图应满足以下特征：

- 纹理丰富
- 局部细节多
- 对比明显
- 不大面积纯色
- 不大面积重复纹理
- 尽量避免镜面高反光设计

### 印刷要求

如果识别图用于实体文创或底座印刷，建议：

- 使用哑光材质优先
- 避免强反光覆膜
- 不要让边框遮挡关键纹理区域
- 保证清晰度不失真

### 交付要求

每张识别图必须同时交付：

- 原始设计图
- 最终导出图 `png`
- 与之对应的 `.mind` 文件

## 5. 3D 模型规范

当前一期框架每个条目默认加载一个主模型：

- 单乐器
- 单乐手
- 小型组合
- 舞台整体模型

## 5.1 存放路径

建议放在：

`public/assets/models/<entryId>/`

示例：

`public/assets/models/cello-memory/scene.glb`

## 5.2 格式要求

推荐优先：

1. `glb`
2. `gltf`

如果使用 `gltf`，要同时交付依赖贴图与二进制资源，不可漏文件。

## 5.3 命名要求

建议统一主文件命名为：

- `scene.glb`
- `scene.gltf`

## 5.4 模型内容要求

模型应满足：

- 中心点合理
- 比例正确
- 不倒置
- 不穿模
- 贴图路径完整
- 可在移动端正常渲染

## 5.5 性能建议

为了适配手机浏览器：

- 单乐器 / 单乐手模型：尽量控制在 `5 万面` 以内
- 小型组合 / 舞台模型：尽量控制在 `15 万面` 以内
- 纹理优先使用 `1024` 或 `2048`
- 尽量减少超大透明材质
- 避免过多独立材质球

## 5.6 动画建议

当前框架允许模型做简单浮动或循环动作，但不强制要求模型自带复杂骨骼动画。

建议优先级：

1. 先保证模型可稳定显示
2. 再考虑简单动画
3. 最后再考虑复杂演奏动作

## 6. 音频素材规范

当前系统支持多条音频分轨同步播放与单轨控制。

## 6.1 存放路径

建议放在：

`public/assets/audio/`

## 6.2 文件格式

一期推荐：

- `wav`

后续如要做线上优化，可统一转为：

- `mp3`
- `m4a`

但同一条目内建议所有音频格式统一。

## 6.3 命名规范

建议格式：

`<entryId>-<stemType>.wav`

例如：

- `cello-memory-melody.wav`
- `cello-memory-harmony.wav`
- `cello-memory-rhythm.wav`

## 6.4 分轨要求

每组分轨必须满足：

- 起点完全对齐
- 总时长一致
- 采样率一致
- 声道设置一致
- 没有额外前导静音

否则前端循环播放时会错位。

## 6.5 推荐交付结构

建议每个条目至少准备 3 条音轨：

1. 主旋律
2. 和声或织体
3. 节奏或支撑层

如果内容更丰富，可扩展到：

- 乐器独奏层
- 合奏总混层
- 环境氛围层

## 6.6 `audioStems` 字段规范

每条音轨包含：

```json
{
  "id": "cello-memory-melody",
  "name": "旋律主线",
  "file": "/assets/audio/cello-memory-melody.wav",
  "defaultEnabled": true,
  "group": "melody",
  "stereoPan": -0.1,
  "gain": 1
}
```

字段说明：

- `id`：唯一标识
- `name`：前端显示名称
- `file`：资源路径
- `defaultEnabled`：默认是否开启
- `group`：分组名称
- `stereoPan`：声像，范围建议 `-1` 到 `1`
- `gain`：音量增益，建议 `0.4` 到 `1`

## 7. 知识内容规范

知识内容通过 `knowledgeCards` 进入系统。

当前系统支持：

- 侧边栏查看全部卡片
- AR 场景映射前 3 张为热点内容

## 7.1 数量建议

每个条目建议至少准备 3 张知识卡：

1. 乐器 / 音色
2. 技法 / 演奏方式
3. 校园叙事 / 文化背景 / 作曲家故事

## 7.2 字段规范

```json
{
  "id": "cello-memory-timbre",
  "anchor": "音色特点",
  "title": "为什么大提琴常被认为最接近人声",
  "summary": "大提琴的音域覆盖较广，中低音区温暖厚实，既能承担旋律也能支撑和声，因此常被用于表达更具歌唱性的段落。",
  "media": []
}
```

字段要求：

- `id`：唯一标识
- `anchor`：按钮与热点标签文字，建议 2–6 字
- `title`：知识卡标题，建议 8–24 字
- `summary`：正文，建议 60–180 字
- `media`：可选扩展媒体

## 7.3 `media` 规范

当前支持的媒体类型：

- `image`
- `audio`
- `link`

示例：

```json
[
  {
    "type": "image",
    "src": "/assets/posters/cello-memory.png",
    "alt": "大提琴主题海报"
  }
]
```

## 7.4 文案要求

知识文案建议遵循以下原则：

- 兼顾专业性与可读性
- 面向非专业学生也能理解
- 避免纯术语堆砌
- 每张卡片只讲一个重点
- 与条目主题高度相关

## 8. NFC / 二维码内容规范

当前项目中的 NFC 与二维码都基于统一深链：

```text
/?entry=<entryId>
```

## 8.1 NFC 内容要求

写入 NFC 标签的内容应为完整 URL，例如：

```text
https://你的正式域名/?entry=cello-memory
```

### 要求

- 必须使用正式 HTTPS 域名
- 必须与条目 `id` 完全一致
- 不要写测试环境地址到正式标签

## 8.2 二维码内容要求

二维码应对应同一条 URL，不要单独维护另一套路由。

二维码可由脚本生成：

```bash
npm run export:links -- --base-url https://你的正式域名
```

导出后查看：

- `generated/nfc-links.json`
- `generated/qrcodes/*.svg`

## 10. 内容提交流程建议

建议内容团队按“条目包”交付，而不是零散交文件。

每个条目包建议包含：

- 条目名称
- 条目 `id`
- 海报文件
- 识别图文件
- `.mind` 文件
- 模型文件
- 音频文件
- 知识卡文案

技术接入前建议先做一次内部自检。

### 自检清单

- `id` 是否唯一
- 所有资源命名是否统一
- 资源路径是否与 `entryId` 对齐
- `.mind` 是否与识别图匹配
- 模型是否能在浏览器中正常加载
- 音频是否时长一致
- 文案是否完成校对
- 二维码 / NFC 深链是否正确

---

## 11. 条目验收建议

每个条目接入后，至少验证以下效果：

- 入口页展示正常
- 深链访问正常
- AR 识别成功
- 模型显示正常
- 音频能正常播放
- 音轨开关与独奏功能正常
- 前 3 个知识热点能正常映射
- 2D fallback 正常显示

---

## 12. 推荐的内容准备优先级

建议按以下顺序推进：

### 第一阶段：最小可用正式内容

先完成 3 个正式条目，每个条目包括：

- 正式海报
- 正式识别图和 `.mind`
- 正式模型
- 3 条分轨音频
- 3 张知识卡

### 第二阶段：试点展示内容

补齐：

- NFC 正式写卡
- 正式二维码物料
- 校园叙事内容
- 展陈说明文案

### 第三阶段：全编制扩展

扩展到 12 个条目，并补充：

- 更丰富的知识卡
- 更精细的模型
- 更完整的分轨系统
- 更完整的校园美育内容体系

---

## 13. 一句话标准

如果一个条目已经具备：

- 明确的 `id`
- 完整的海报
- 可识别的识别图与 `.mind`
- 可加载的模型
- 可同步的分轨音频
- 可展示的知识卡

那么它就已经符合当前框架的一期接入标准。
